# 原始材料面板 — 分步落地实施指南

> 给 Sharon：这份文档基于你的 `原始材料面板 - 优化原型.html`、`知识库前端优化.md` 和 `feat/wiki-raw-material-panel` 分支整理。
> 结论先说：**你的原型设计整体成立，直接作为这个功能的蓝图使用**。目前唯一的结构性问题是前端走在了后端前面——原型里的路径分组、独立扫描、定时任务都需要后端有"扫描源"这个概念，而现在还没有。所以这份文档把工作切成三步：能立刻合并的先合（PR-A），后端地基补上（PR-B），最后接线让原型完整落地（PR-C）。

## 0. 现状盘点：后端已经有什么（不要重造）

| 能力 | 位置 | 说明 |
|---|---|---|
| 多路径 + glob 扫描 | `WikiDirectoryScanService` | `sourceDirectory` 已支持**换行分隔的多路径/glob**（如 `/data/ocr/**/*.txt`）；显式扩展名的 glob 会跳过默认扩展名过滤 |
| 路径安全校验 | `WikiSourcePathValidator` | 符号链接逃逸检查、路径合法性；**任何新的扫描入口必须复用它** |
| 扫描去重 | `WikiDirectoryScanService` | 按解析后的绝对路径去重，已存在的文件跳过——**这就是"增量扫描"，无需新实现** |
| 定时自动同步 | `WikiSourceWatcherService` | KB 级 watcher（全局开关 + 每 KB 开关），基于 ShedLock（JDBC 分布式锁，无 Redis） |
| 手动扫描/设目录 API | `PUT .../source-directory`、`POST .../scan` | 前端调用在 `mateclaw-ui/src/api/index.ts:819-821` |
| 页面反查 | `useWikiStore.filterPagesByRaw` | 原型预览弹窗里"查看生成的页面 →"可以直接用它实现 |

**关键推论**：原型弹窗里的「服务器物理路径 + 文件类型过滤」两个字段，今天就能落到真实后端（路径写入 `sourceDirectory` 多行文本，过滤用 glob 表达）。真正缺的只有三样：**路径的"身份"（别名）、材料的"归属"（source_id）、每路径的定时配置**。这就是 PR-B 的全部范围。

### 2.3 顺手修正的四个点

1. **预览大小上限**：`openPreview` 里 `blob.text()` 之前检查 `raw.fileSize`，超过 2MB 显示"文件过大，请下载查看"（复用 `preview.binaryHint` 的展示位新增一条文案）。否则大 CSV 会把整页渲染卡死。
2. **批量重处理的刷新风暴**：`batchReprocess` 循环里每次 `reprocess()` 都会触发 1 次立即 + 2 次延迟的全列表刷新（选 20 个 = 60 次请求）。改法：循环里只调 `wikiApi.reprocessRaw`，循环结束后统一 `fetchRawMaterials` 一次 + 一组延迟刷新。
3. **两处文案**：筛选栏的 `sources.sourceType` 现在是"路径"，但它筛的是来源类型（UPLOAD/TEXT/SCAN），改成"来源类型"；`batchReprocess` 的"批量重扫"实际动作是重新处理（不触发扫描），改成"批量重新处理"，避免和 PR-C 的"扫描"概念打架。
4. **计数口径**：状态 chip 的"失败"计了 `failed + cancelled`，但异常分区只收 `failed`，两个数字对不上。统一为：chip"失败"只计 `failed`（cancelled 归"全部"里即可）。

## 3. PR-B：后端规格 — `wiki_kb_source`

> **上游 issue 已建：[#487](https://github.com/mateaix/mateclaw/issues/487)**，正文规格与本节一致（你的原型和需求文档就是它的设计依据）。动手前看一眼 issue 下维护者的反馈，有方向性意见先消化进设计再写代码。
> 迁移脚本要在**所有方言目录**同版本号同步新增（当前 `h2` / `mysql` / `kingbase`，若 PG 树已合入则一并；写作时 upstream dev 已到 V166，实施时以当时最新为准）。四套方言的坑（KingbaseES 用 `SMALLINT` 逻辑删除、JSON 存 `TEXT` 等）不熟的话直接找倪程伟对一遍，这块他踩过全部的坑。

### 3.1 表：`wiki_kb_source`

对齐你前端 `PathConfig` 接口的字段命名（`RawMaterialPanel.vue:861-869`），接线时基本是机械替换：

| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT（雪花） | |
| `kb_id` | BIGINT | 所属知识库 |
| `alias` | VARCHAR(128) | 业务别名（"中基协2025处罚公告"）；同 KB 内唯一 |
| `path` | VARCHAR(1024) | **单条**路径或 glob（多路径 = 多行 source 记录，别再塞换行分隔） |
| `file_filter` | VARCHAR(256) | 可空；逗号分隔 glob（`*.pdf, *.txt`），实现上合成到 path 的 glob 匹配里 |
| `cron` | VARCHAR(64) | 可空；空 = 不定时。预设值直接存 cron 表达式（`0 0 2 * * ?` 等） |
| `enabled` | TINYINT/SMALLINT | 定时开关 |
| `create_time` / `update_time` / `deleted` | | 仓库通用约定 |

**两条硬约束（设计阶段就定死，这是复审时发现的两个坑）：**

1. **定时任务只做增量，没有"定时全量"**。全量重扫 = 手动动作（扫描 + 对该 source 下已有材料批量 reprocess）。原因：108 个 PDF 每天 02:00 全量过一遍 LLM 消化是纯烧 token，你原型里排期文案写的也是"按计划自动**增量**扫描"——直觉是对的，schema 层面把它固化。所以**不需要 `scan_mode` 列**，模式只是手动扫描接口的一个参数。
2. **出处与分组分离**。`wiki_raw_material` 加两列：
   - `source_id`（可空，**不可变**）：记录文件从哪个 source 扫来的，是客观出处；手动上传/粘贴为 NULL；
   - `group_override_id`（可空）：你需求文档里的"迁移分组"改的是这一列，展示分组 = `COALESCE(group_override_id, source_id)`。
   为什么不能只用一列：如果迁移直接改 `source_id`，出处信息被销毁，而且增量扫描按路径去重时会和"文件已被迁走"产生歧义。分开之后，迁移不影响下次扫描的去重判断。

### 3.2 API（挂在现有 wiki controller 风格下）

```
GET    /api/v1/wiki/knowledge-bases/{kbId}/sources           列表（含每 source 文件数统计）
POST   /api/v1/wiki/knowledge-bases/{kbId}/sources           创建（路径过 WikiSourcePathValidator）
PUT    /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}     更新
DELETE /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}     删除（材料保留，source_id 置空归入未分组）
POST   /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}/scan?mode=incr|full   单路径扫描
PUT    /api/v1/wiki/knowledge-bases/{kbId}/raws/group        批量迁移分组 {rawIds, targetSourceId}
```

- 权限：全部走 `manage:wiki`（和现有扫描一致）
- 扫描落库时写 `source_id`；raw material 列表 DTO 增加 `sourceId` / `groupOverrideId`
- 定时：扩展 `WikiSourceWatcherService`（或并列新 job），ShedLock 锁粒度按 source；全局 watcher 开关继续作为总闸
- 兼容：存量 KB 的 `sourceDirectory` 保持可用；迁移脚本把非空的 `sourceDirectory` 按行拆成 source 记录（alias 默认"默认扫描路径 N"），`sourceDirectory` 字段标记废弃但不删
