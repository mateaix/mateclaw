## 0. 现状盘点：后端已经有什么（不要重造）

| 能力 | 位置 | 说明 |
|---|---|---|
| 多路径 + glob 扫描 | `WikiDirectoryScanService` | `sourceDirectory` 已支持**换行分隔的多路径/glob**（如 `/data/ocr/**/*.txt`）；显式扩展名的 glob 会跳过默认扩展名过滤 |
| 路径安全校验 | `WikiSourcePathValidator` | 符号链接逃逸检查、路径合法性；**任何新的扫描入口必须复用它** |
| 扫描去重 | `WikiDirectoryScanService` | 按解析后的绝对路径去重，已存在的文件跳过——**这就是"增量扫描"，无需新实现** |
| 定时自动同步 | `WikiSourceWatcherService` | KB 级 watcher（全局开关 + 每 KB 开关），基于 ShedLock（JDBC 分布式锁，无 Redis） |
| 手动扫描/设目录 API | `PUT .../source-directory`、`POST .../scan` | 前端调用在 `mateclaw-ui/src/api/index.ts:819-821` |
| 页面反查 | `useWikiStore.filterPagesByRaw` | 原型预览弹窗里"查看生成的页面 →"可以直接用它实现 |

**关键推论**：原型弹窗里的「服务器物理路径 + 文件类型过滤」两个字段，今天就能落到真实后端（路径写入 `sourceDirectory` 多行文本，过滤用 glob 表达）。真正缺的只有三样：**路径的"身份"（别名）、材料的"归属"（source_id）、每路径的定时配置**。这就是 PR-B 的全部范围。

4. **计数口径**：状态 chip 的"失败"计了 `failed + cancelled`，但异常分区只收 `failed`，两个数字对不上。统一为：chip"失败"只计 `failed`（cancelled 归"全部"里即可）。

> **2026-07-04 更新：这一节你已经用行动超越了。** `8d0ac970` 把 PR-B 实现出来了（`mate_wiki_source_group`），而且有两处比本节原规格做得更好——详见 §3.4 的评审。§3.1–§3.3 保留为原始规格存档，**以你的实际实现为准**；上游 issue [#487](https://github.com/mateaix/mateclaw/issues/487) 我们会补一条 as-built 评论同步差异，上游按你的实现评审而不是按旧规格。

### 3.1 表：`wiki_kb_source`

对齐你前端 `PathConfig` 接口的字段命名（`RawMaterialPanel.vue:861-869`），接线时基本是机械替换：

| 列 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT（雪花） | |
| `kb_id` | BIGINT | 所属知识库 |
| `alias` | VARCHAR(128) | 业务别名（"中基协2025处罚公告"）；同 KB 内唯一 |
| `path` | VARCHAR(1024) | **单条**路径或 glob（多路径 = 多行 source 记录，别再塞换行分隔） |
| `file_filter` | VARCHAR(256) | 可空；逗号分隔 glob（`*.pdf, *.txt`），实现上合成到 path 的 glob 匹配里 |
| `cron` | VARCHAR(64) | 可空；空 = 不定时。预设值直接存 cron 表达式（`0 0 2 * * ?` 等） |
| `enabled` | TINYINT/SMALLINT | 定时开关 |
| `create_time` / `update_time` / `deleted` | | 仓库通用约定 |

**两条硬约束（设计阶段就定死，这是复审时发现的两个坑）：**

1. **定时任务只做增量，没有"定时全量"**。全量重扫 = 手动动作（扫描 + 对该 source 下已有材料批量 reprocess）。原因：108 个 PDF 每天 02:00 全量过一遍 LLM 消化是纯烧 token，你原型里排期文案写的也是"按计划自动**增量**扫描"——直觉是对的，schema 层面把它固化。所以**不需要 `scan_mode` 列**，模式只是手动扫描接口的一个参数。
2. **出处与分组分离**。`wiki_raw_material` 加两列：
   - `source_id`（可空，**不可变**）：记录文件从哪个 source 扫来的，是客观出处；手动上传/粘贴为 NULL；
   - `group_override_id`（可空）：你需求文档里的"迁移分组"改的是这一列，展示分组 = `COALESCE(group_override_id, source_id)`。
   为什么不能只用一列：如果迁移直接改 `source_id`，出处信息被销毁，而且增量扫描按路径去重时会和"文件已被迁走"产生歧义。分开之后，迁移不影响下次扫描的去重判断。

### 3.2 API（挂在现有 wiki controller 风格下）

```
GET    /api/v1/wiki/knowledge-bases/{kbId}/sources           列表（含每 source 文件数统计）
POST   /api/v1/wiki/knowledge-bases/{kbId}/sources           创建（路径过 WikiSourcePathValidator）
PUT    /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}     更新
DELETE /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}     删除（材料保留，source_id 置空归入未分组）
POST   /api/v1/wiki/knowledge-bases/{kbId}/sources/{sid}/scan?mode=incr|full   单路径扫描
PUT    /api/v1/wiki/knowledge-bases/{kbId}/raws/group        批量迁移分组 {rawIds, targetSourceId}
```

- 权限：全部走 `manage:wiki`（和现有扫描一致）
- 扫描落库时写 `source_id`；raw material 列表 DTO 增加 `sourceId` / `groupOverrideId`
- 定时：扩展 `WikiSourceWatcherService`（或并列新 job），ShedLock 锁粒度按 source；全局 watcher 开关继续作为总闸
- 兼容：存量 KB 的 `sourceDirectory` 保持可用；迁移脚本把非空的 `sourceDirectory` 按行拆成 source 记录（alias 默认"默认扫描路径 N"），`sourceDirectory` 字段标记废弃但不删

### 3.4 `8d0ac970` 评审意见（2026-07-04）

先说整体：**这个提交的质量和对仓库惯例的把握，完全不像第一次动这个模块后端的人写的。** 几处具体值得点名——

- **回填策略是全提交最成熟的工程判断**：历史材料的 `groupId` 不做一次性路径匹配脚本（你注释里写的"风险高、容易匹配错"是对的），而是留给分组下一次正常扫描自然补写。稳、可解释、天然幂等。
- **你对 issue #487 的偏离是一个更好的简化**：原规格用两列（`source_id` 不可变 + `group_override_id`）解决"迁移分组 vs 扫描去重"的冲突，你用单列 `groupId` + 一条规则替代——`tagGroupIfNeeded` 只在 null 时补写、永不覆盖手动改挂。手动迁移因此是"粘性"的，而且你专门写了 `incrementalScan_skipHitAlreadyGroupedNotRetagged` 把这条规则锁进测试。代价只是丢掉"文件最初从哪来"的出处记录，v1 完全可接受。**这一条会同步进 issue #487。**
- IDOR 测试照 `WikiEntityControllerIdorTest` 的模式六个端点全覆盖；MySQL 迁移用 `INFORMATION_SCHEMA` + prepared statement 幂等且注释里引了 V146/V161 先例；partial-PUT 只覆盖非 null 字段正是 #93/#477 沉淀的惯例；`countRawByGroup` 一次聚合避开 N+1。这些都说明你把仓库的历史读进去了。

以下按优先级分级，前两条是合并门槛，其余不阻塞。

**必须修（合并前）：**

1. **迁移版本号撞车**：V166/V167 基于 fork dev（当时最新 V165）取号，但 upstream dev 已有自己的 V166。下次同步上游 Flyway 会因重复版本号启动失败。修法：先合入 upstream/dev，再把两个迁移文件重命名到当时空闲的版本号（三套方言同步改）。如果你没处理过 upstream 同步的冲突，这次 merge 可以由倪程伟来做，你只管重命名。
2. **`WikiSourceGroupEntity.deleted` 缺 `@TableLogic`**：wiki 模块所有带 `deleted` 列的实体都是逻辑删除，你的实体是裸字段——`deleteById` 目前走物理删除、查询也不过滤。加上注解，语义自动对齐模块约定。

**应该修（同 PR 内小改动）：**

3. `scan_mode` 列与 `?mode=` 参数职责重叠：扫描端点完全不读 `group.scanMode`，这列现在只是被存着。建议直接删列（模式只作请求参数，顺带消灭将来"定时全量烧 token"的隐患）；若保留，则参数缺省时回落到列值。
4. `cronExpr` 落库无校验：创建/更新时对非空值跑一遍 `CronExpression.parse()`，坏表达式当场 400，别留给将来的调度器炸。调度器落地时执行既定约束：**cron 触发只允许增量**。
5. `(kb_id, alias)` 缺唯一索引，create 也不查重——同 KB 可建两个同名分组。注意加了 `@TableLogic` 之后唯一索引要考虑软删行，三套方言写法不同，这块可以找倪程伟对。
6. `updateRawGroupBatch` 的归属过滤：`listByKbId` 拉全表再 `contains` 是 O(n·m)，换成 `selectList(in(id, rawIds).eq(kbId))` 一次查询。这段防跨库越权的过滤逻辑本身是对的，但**没有测试**——补一个：批量请求里混入外库 rawId，断言实际 update 的 id 集合不含它（断言落在绑定参数上，仓库惯例）。

**可以讨论（不阻塞）：**

7. raw 列表新增返回 `sourcePath`（服务器绝对路径），端点 viewer 可读——确认是否有意暴露。
8. 全量扫描对命中路径下所有 skip-hit 强制重跑，不看 `raw.groupId` 是否属于本组：手动迁到 B 组的文件，A 组全量扫描仍会重跑它（不改归属，只重跑管道）。内容上说得通，语义上略反直觉——至少接口文档写明，或 force 循环里过滤 `groupId == group.id`。
9. 两处陈旧注释顺手更正：`WikiRawMaterialEntity.sourcePath` 还写着 "binary uploads only"（你的打标机制正依赖文本扫描也写它）；`fileFilter`/`cronExpr` 在 partial-PUT 语义下无法清空回 null——这和 #484 讨论的"配置键删除语义"是同一个坑，不必本 PR 解决，挂到那边统一处理。

**最值得补的一个测试：**

`updateGroup` 的 javadoc 里你自己写明了那个坑——"必须用 `LambdaUpdateWrapper.set` 显式赋值，否则 NOT_NULL 策略下无法把 groupId 置空"——但恰恰没有测试验证它（服务层测试把 mapper 全 mock 掉了）。补一个 H2 落库的小测试：设组 → 清空为 null → 断言真的变 null。它锁的是你自己发现的框架陷阱，也是全提交里最有长期价值的一个测试。
