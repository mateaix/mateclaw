## 1. PR-1：后端（`8d0ac970` + `8a004435` 后端部分，新分支提交）

**合并门槛：**

1. **迁移重编号**（上轮遗留，这轮已被实测证实）：跑测试时环境里恰好同时出现了 upstream 的 `V166__message_usage_detail.sql` 和你的 `V166__wiki_source_group.sql`，Flyway 当场 `Found more than one migration with version 166`、应用起不来——合并 upstream 后每台机器都会这样。先合入 upstream/dev（这次 merge 可以由倪程伟做），再把 V166/V167 重命名到空闲版本号，三套方言同步。
2. **修上面第 0 节的两个编译错。**
3. **软删 × 唯一索引的组合坑**：删除分组后，同名别名**永远建不回来**——`assertAliasAvailable` 被 `@TableLogic` 过滤看不到软删行 → 查重通过 → INSERT 撞唯一索引 → 裸 500（不是友好 400）。而"删掉分组再建同名的"是你自己场景里的常见操作。说句公道话：这个坑是评审建议（加 @TableLogic + 加唯一索引）组合出来的，不是你的失误。修法：`delete()` 里先把 alias 改成墓碑名（如 `alias + "#del#" + id`）再逻辑删，唯一性、软删约定、可重建三样全保住；补一个测试：删组 → 同名重建 → 成功。
4. **让 `fileFilter` 真正生效（对抗式审查发现，集体盲区）**：这个字段现在是"存了、展示了、但扫描不用"——`WikiDirectoryScanService` 里对它的引用是零，用户配 `*.pdf` 扫描照样全吃。你前后端的管道都铺好了，只差最后一厘米：在 `scanGroup` 里把 filter 合成进已有的 PathMatcher glob 匹配（后端机制是现成的），补一两个测试；如果想留到后续 PR，也可以先把表单字段藏掉——唯一不行的是让 UI 承诺一个不存在的功能。

**同 PR 小改：**

5. `updateRawGroupBatch` 归属过滤（上轮遗留）：`listByKbId` 拉全表再 `contains` 是 O(n·m)，改成 `selectList(in(id, rawIds).eq(kbId))` 一次查询；补测试：批量请求混入外库 rawId，断言实际 update 的 id 集合不含它（断言落在绑定参数上）。
6. **E2E 测试改连内存库**：`WikiRawMaterialGroupE2ETest` 现在连的是 `data/` 下共享持久 H2——每跑一次就往开发库写测试行，还依赖本地库状态。在 `@SpringBootTest` properties 里覆写 `spring.datasource.url=jdbc:h2:mem:...`。
7. **全量扫描的强制重跑循环加两个过滤**（对抗式审查发现）：一是跳过 `processing`/`pending` 状态的 raw——行级重处理按钮在前端对处理中的行是隐藏的，但全量扫描绕过了这层守卫，可能把管道里跑到一半的材料重新置 pending 引发并发处理；二是跳过已被手动迁到其它组的文件（`groupId != group.id`，上轮讨论项升级）。两个条件放同一个 if 里就行。

**可讨论（不阻塞，PR 描述里提一句）**：raw 列表向 viewer 返回 `sourcePath`（服务器绝对路径）是否有意；`WikiRawMaterialEntity.sourcePath` 的 "binary uploads only" 注释已过时；建议在 PR 描述里提醒部署方配置 `mate.wiki.allowed-source-roots`——路径校验默认 fail-open 是平台既有姿态（桌面/单租户默认），但你的新 UI 让"任何 member 都能配服务器路径"这件事可发现性大增，多租户部署应该收紧。

## 2. PR-2：前端整体（`4705c077` + `8a004435` 前端部分，依赖 PR-1 先合）

原 PR-A 的四个小修（仍然有效）：

1. 预览加大小上限：`fileSize` 超 2MB 显示"文件过大，请下载查看"，否则大 CSV 卡死页面。
2. `batchReprocess` 循环里只调 API，结束后统一刷新一次（现在选 20 个会触发 60 次列表刷新）。
3. 文案：筛选栏"路径"→"来源类型"；"批量重扫"→"批量重新处理"（它不触发扫描）。
4. 计数口径：状态 chip"失败"只计 `failed`，与异常分区对齐。

这轮新增的四个小修：

5. **全量扫描加确认弹窗**（`mcConfirm`）：现在误点一次分组头的全量按钮 = 整组文件重新过 LLM，没有任何拦截。
6. **`weekly1` 预设表达式与文案不一致**：`CRON_PRESETS.weekly1 = '0 0 2 * * MON'`（02:00），文案写"每周一 03:00"——改其一。
7. **fileFilter 清空无效**：`form.fileFilter.trim() || null` 把空串转成 null，而 null 在后端 partial-PUT 语义下是"不改"。清空过滤器这个操作会静默失败——直接发 `form.fileFilter.trim()`（空字符串），后端非 null 即覆写。
8. **中文 cron 占位符的示例还是 5 段**（对抗式审查发现）：`zh-CN` 的 `cronCustomPlaceholder` 是 `'例：0 2 * * *（秒 分 时 日 月 周）'`——示例 5 段、说明 6 段，用户照抄示例会吃 400。英文版你已经改对了，中文补上 `0 0 2 * * *` 即可。

收尾清理：

9. 删掉旧的目录输入框 + 扫描行相关的孤儿 i18n 键（`wiki.scan/scanning/scanResult/dirPlaceholder` 等）——每路径扫描已经落地，过渡入口不再需要。
10. `pnpm run build` + `pnpm run lint`；中英 i18n 键对齐检查。

**验收**：对照 `原始材料面板 - 优化原型.html` 逐区块走查——路径配置、分组折叠、单路径扫描（增量/全量）、定时开关与"未生效"徽章、迁移分组（单条 + 批量）、异常分区、预览。原型即验收标准。
